# 数据源概览

## 一、整体概况

本章节主要从数据集市角度介绍数据源相关信息，重点介绍数据源相关的库表。同时本章节也会介绍数据源相关的一些基本概念、集市表概况，希望读者通过本章节的内容能够快速找到想要的数据源相关库表，能够快速响应取数和分析需求，提高工作效率。

### 1、数据范围

本章节介绍的数据源主要包含两类，一类为我行APP、公众号、小程序等应用采集到的数据；一类为通过征信平台、数管等系统从供应商处获取的外部数据源，如税银、人行、司法、汇法等。

### 2、基础概念

#### 1. 数据源与数据源接口

我行风险部门可以根据业务场景与业务需求引入外部信息来丰富客户征信信息，而外部信息一般是通过数据源服务商提供的，一个服务商通常对应一个数据源。而一个服务商通常提供多种数据，每种数据通常对应一个数据源接口。例如通过中国司法大数据研究院有限公司引入的“司法大数据”数据源，共包含3个数据源接口：限高、失信和定制接口。

#### 2. 流水表、报告表、数据表

通常接入一个数据源后，征信平台会存储3类表：

**报告表：**每个数据源有一个主报告表，报告表表名中通常包含关键字 report，一般情况同一个数据源的多个接口会共用一个报告表。

**流水表：**每次查询数据源接口都会有一条流水记录，记录是否查询成功、查询参数、是否查询缓存，流水表表名通常包含关键字 trans。如果接口返回结果比较简单一般只有一张表名包含 trans_info 的表，会记录所有查询记录（包括外查和查询缓存）。如果接口比较复杂，可能还有一张表名为 trans_detail 的表，trans_detail 一般仅包括外查的记录。

**数据表：**数据源接口返回的原始报文通常是 json 格式的，征信平台会解析原始报文，并将结果存放在数据表中，如果接口返回的报文比较复杂，会存放在多个数据表中；如果报文比较简单，也可能没有数据表，直接将报文结果存放在报告表中。

#### 3. 流水号、报告号

征信平台外查数据源都会自动生成一个唯一报告号（字段名一般为 order_no 或 reportsn），用来标识本次查询信息。如果后续再次查询该客户的征信信息且命中缓存规则，则返回的是同一个报告号。同时贷前/贷后各系统每次调用征信平台时请求参数中都有业务流水号（对应流水表中的 biz_seqno 字段），一般通过该流水号可以将征信平台的数据与其他系统的数据关联起来。但需注意，征信平台只在流水表中登记该流水号，并未校验流水号是否重复。

#### 4. trans_code、report_code

transcode 是征信平台定义的产品查询征信源标识，征信产品不管走什么渠道 transcode 都是唯一确定值。transcode 由两部分信息组成：source_name（征信源名称）和 report_code（征信源接口编码），如：FICO_SCORE, FICO为 source_name, SCORE为 report_code。

### 3、集市数据源整体框架

#### 1. 拆表逻辑

征信平台包含了全行的征信数据，按照数据治理规范要求，同步到企金风险库中的数据只能是企金条线产品的数据，因此征信平台会将数据按产品对口业务部门在BDP中将表拆分为 dcb、daf、bibf 后缀的表。而企金风险部主要负责后缀为 dcb、bibf 的数据，因此一般情况同一张表在BDAP会对应两张表，分别存放 dcb、bibf 条线的数据。

#### 2. 分层框架

当前业务同事经常直接使用从征信平台同步到BDAP的原始ODS表，但使用这些表存在以下痛点：

```Plaintext
* 原始字段晦涩难懂，需要借助接口文档才能理解字段含义
* 原始表中缺乏 ccif/ecif 等关键字段，需业务同学自己关联，关联逻辑复杂，使用门槛高
* 业务同学使用原始ODS表，各自加工加工数据，使用效率较低
* 公共逻辑重复加工、浪费计算和存储资源
```

为了解决上述问题，中台集市正在分数据源逐步建设明细层，以解决上述痛点，具体方案包括：

```Plaintext
* 对维度值进行统一映射，字段清晰易懂
* 建设mapping表，明细表中加入 ccif/ecif 等关键信息
* 建设统一的明细层，将加工逻辑做到明细层，做到使用简单
* 统一调度，避免重复建设导致资源浪费
```

其中每个数据源的明细层均包括：

**snap表**，包括两类：一类是直接对ods表合并去重，所有字段和ods表保持一致，表名一般为 snap+ods 表名；一类是包含 ccif 信息的表，这些表经过 ccif mapping 处理，可以直接通过客户号找到报告信息，表名一般为 snap+ods表名+ccif。

**dwd表**，通常将同一数据源接口返回的相同粒度的数据整合在一张表中，同时将枚举值中文含义等信息也添加到dwd表中，能够帮助大家快速理解字段含义，方便使用。

### 4、如何快速找到想要的表

#### 1. 如何查找征信平台的表

目前征信平台同步到BDAP的表超过2000张，可按如下步骤尝试找到想要的表：

①首先明确要找的数据源接口名称，可以从征信平台apidesign目录页（[征信数据源-目录](http://apidesign.weoa.com/apidesign-core/#/project/5929/docs?catId=666bae3b7739cc179b44abc7&title=征信数据源接口--目录)） 通过关键字搜索到对应的接口。比如查询汇法企业接口，可以直接搜“汇法”关键字：

![image-20260214131908247](C:\Users\m8705\AppData\Roaming\Typora\typora-user-images\image-20260214131908247.png)



②找到该接口对应的 transcode，如果目录页没有登记 transcode，可以打开对应的apidesign详情页查看，比如汇法企业的 transcode 是“LAW_ENT_PER_V1”。 ③transcode 中包含的关键字也是表名中的关键字，可以通过 transcode 中的部分关键字，去 brm_wyd_ods_mask 库中搜索表名，比如我们用“LAW”作为关键字，在库中搜索表名：

![image-20260214131858825](C:\Users\m8705\AppData\Roaming\Typora\typora-user-images\image-20260214131858825.png)



④一般情况可以搜索到多个表名，此时需要从中选择最可能的表，对比该表的字段名与接口文档中介绍的字段，如果能够匹配，则说明找到了正确的表，如果不匹配，则需要继续查找其他表。

#### 2. 如何查找流水表

流水表中有 source_name、report_code 字段（字段名称可能略有差异），可以直接拼接成 transcode，一般先利用关键字找到可能的流水表，然后根据 source_name、report_code 来查找表中是否有该 transcode 的数据，来验证找到的 trans 表是否准确。
