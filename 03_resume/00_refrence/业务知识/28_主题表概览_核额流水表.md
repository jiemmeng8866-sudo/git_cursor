# 核额流水表

##### 核额流水表：brm_ent_dm_work.dwd_marm_quota_req_flow_ds 【dw1040表】

## 1 背景

**业务背景：**银行根据客户的贷款申请对客户的贷款额度进行限额审查，并核验更新客户的可贷款额度；具体可分为首次核额和定期核额，考察指标为核额金额、核额各规则下对应的通过率；客户申请多个产品，则会出现多条核额流水号。若多产品间存在互斥类或合并类，仅存在单一产品对应的额度quota大于0；

**订货贷核额场景说明：**V2升级至V3，订货贷-订调整即不再支持品牌层额度，仅有产品层额度；因此，订货贷含品牌场景，其核额流水表内的quota为0可能标识其对应的品牌额度为0，但产品层仍有额度；**（注：不再使用品牌额度，产品额度共享，自主、受托由统一****授信****做限额管控，若想取详细的产品/品牌额度，可参考常用字段介绍-quota）**

**集市核额流水表粒度为quota_req_flow_busi_sno（业务流水号）；**该表以ods_marm_quota_req_flow表为主表关联各种产品的核额引擎的出入参信息及产额度来源和额度划分等信息，可用于观察借据的核额流水信息及额度来源等信息，进一步监测规则命中和模型通过等贷前通过率指标波动是否存在异常。

**➤** ***科创贷申请流程：***

Step1：is_risk_apply = 1 是否申请

Step2：is_allow_in = 1 是否准入

Step3：is_pass_rule = 1 是否通过规则

Step4：kcd_is_pass_first_approval = 1 是否通过初审Step5：is_obtain_quota = 1 是否获得授信（是否通过终审），获得额度为nvl(kcd_obtain_quota,quota)

## ✍ 常见问题

**（1）为什么较多借据关联的核额流水的核额状态是失败?**

经排查，得出如下几个原因：

1. 借新还旧、无还本续贷、短贷续贷导致新借据仍对应原借据的核额流水原先就为0（部分原因）；
2. 订货贷含品牌类产品，其核额流水的quota为0可能标识其对应品牌的额度为0，但产品有额度（占比较大）；
3. 个人经营贷升级至企业贷，产品升级后，贷前将历史核额记录的产品ID重新为核额成功的产品（可能非借据产品），导致借据产品与核额成功产品不对应；
4. 产品升级（举例下图）：V1时期科创贷有额度，微业贷贷前核额有额度但由于额度切分导致实际核额为0，由于产品升级（V1至V2），科创贷额度下调，基于max(额度)规则分配至额度最大的产品微业贷（升级不会影响额度变化），由于底表只刷新了统一授信协议对应的授信额度，并没有统一核额流水信息，所以出现线上核额流水中有借据对应产品号1010013是无核额的现象，但实际下发是科创贷的额度[可结合统一授信信息来看]；

目前，综上评估剔除前三个问题的核额失败的现象占比小于1%，目前短时间还无法调整口径。

**（2）核额表中的risk_prod_type字段为null的字段含义?**

risk_prod_type是常用分析标签、风险模型大类标签，集市报表体系常用口径。

核心逻辑：按risk_prod_detai向上汇总，因此存在一些条件限定(场景)导致为空，例如是否下发，科创所处阶段等，也包含历史已下限产品（如供应链金服融资、微保贷等）及商E贷、南山上市促进贷等；

提示：若要计算策略规则通过率，则取数时需限制risk_prod_type is not null

**（3）存在大约五千条借据关联不到核额流水号且此类借据risk_prod_type字段为null的原因？**

经排查，得出如下几个原因：

**（4）核额流水号的继承情况？**

1. 组合发起核额，组合生成busi_sno给贷前，贷前核额后传给组合，组合再下发到统一授信bizno，公贷获取统一授信bizno保存至借据treatybizno。
2. 组合credit_faciltity_log.busi_sno  == 核额quota_req_flow.busi_sno ==  统一授信credit_treaty.bizno == 借据business_duebill.treatybizno
3. 风险直接发起核额，贷前核额生成busi_sno，传给组合，组合下发到统一授信bizno，公贷获取统一授信bizno保存在treatybizno。
4. 核额quota_req_flow.busi_sno == 组合credit_faciltity_log.busi_sno  ==  统一授信credit_treaty.bizno == 借据business_duebill.treatybizno
5. 企业经营贷一转二，贷前核额生成busi_sno，传给组合，组合拆分成两个产品，下发给统一授信两条记录：不同的产品ID+相同的busi_sno，统一授信两条记录为：不同的产品ID+相同的bizno，公贷获取统一授信bizno保存在treatybizno。
6. 核额quota_req_flow.busi_sno == 组合credit_faciltity_log.busi_sno *2  ==  统一授信credit_treaty.bizno *2  == 借据business_duebill.treatybizno *2
